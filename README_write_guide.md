# README 작성 가이드 (README_write_guide)

이 문서는 이 프로젝트의 `README.md`를 추후 업데이트할 때 지켜야 할 작성 규칙을 정의한다.

## 1. 목적

- 처음 보는 사람이 **30초 안에 프로젝트 성격**을 이해하도록 작성한다.
- 실행 방법, 핵심 기능, 구조를 빠르게 찾을 수 있게 한다.
- 불필요한 템플릿 문구(CRA 기본 장문 등)는 넣지 않는다.

## 2. 기본 원칙

- 한국어를 기본으로 작성한다.
- 문장은 짧고 명확하게 쓴다.
- 사실만 작성한다. 코드/설정에 없는 내용은 추측해서 쓰지 않는다.
- 섹션 순서와 제목은 가급적 고정한다.
- 코드 블록은 반드시 언어 태그를 붙인다(`bash`, `text` 등).

## 3. 권장 섹션 순서

아래 순서를 기본 템플릿으로 사용한다.

1. 프로젝트명
2. 한 줄 소개
3. 주요 기능
4. 기술 스택
5. 시작하기
   - 의존성 설치
   - 로컬 실행
6. 스크립트
7. 프로젝트 구조
8. 라우트(해당 시)
9. 배포(해당 시)
10. 문서/트러블슈팅(해당 시)
11. 부가 정보(선택)
   - 스크린샷 / 데모 URL / 데이터 출처 / 기여 방법 / 라이선스

## 4. 섹션별 작성 규칙

### 4.1 한 줄 소개
- "무엇을 위한 프로젝트인지"를 한 문장으로 작성한다.
- 기술명 나열보다 사용자 관점 목적을 우선한다.

### 4.2 주요 기능
- 불릿 리스트로 작성한다.
- 기능명은 명사형/동사형 혼용 없이 통일한다.
- 실제 동작하는 기능만 적는다.

### 4.3 기술 스택
- 핵심 스택만 기재한다(예: React, TypeScript, Router, 스타일링 도구).
- 버전 표기는 `package.json` 근거가 있을 때만 명시한다.

### 4.4 시작하기/스크립트
- 실행 가능한 명령어만 넣는다.
- 패키지 매니저는 현재 프로젝트 기준(`yarn`)을 따른다.
- 명령어 옆에 한 줄 설명을 붙인다.

### 4.5 프로젝트 구조
- 최상위 핵심 디렉터리만 보여준다.
- 주석으로 역할을 간단히 명시한다.

### 4.6 라우트
- `경로 - 설명` 형태로 통일한다.
- 앱 라우트 정의(`src/App.tsx`)와 불일치하면 안 된다.

### 4.7 배포/환경변수
- production URL, 배포 플랫폼, 필수 env var를 현재 설정 기준으로 적는다.
- secret 값은 절대 적지 않고 변수명만 적는다.
- Vercel Function이나 rewrite처럼 운영 장애와 직결되는 설정은 파일 경로와 함께 적는다.

### 4.8 문서/트러블슈팅
- `docs/troubleshooting/`에 재발 가능성이 있는 장애 기록이 있으면 README에서 링크한다.
- blueprint/template 문서는 현재 앱 기능과 구분되도록 목적을 짧게 설명한다.

## 5. 금지 사항

- 확인되지 않은 성능 수치, 사용자 수, 운영 환경 정보를 기재하지 않는다.
- 현재 프로젝트와 무관한 일반 문서를 길게 복붙하지 않는다.
- 문서만 화려하고 실행/기능 정보가 없는 상태를 만들지 않는다.

## 6. 업데이트 체크리스트

README 수정 전/후 아래 항목을 확인한다.

- [ ] 프로젝트명과 한 줄 소개가 현재 서비스 목적과 일치하는가?
- [ ] 주요 기능 목록이 실제 코드와 일치하는가?
- [ ] 실행 명령어(`yarn start`, `yarn build`, `yarn test`)가 동작 가능한가?
- [ ] 라우트 목록이 최신 상태인가?
- [ ] 배포 URL, Vercel 설정, 필수 env var가 최신 상태인가?
- [ ] troubleshooting 문서 목록이 최신 상태인가?
- [ ] 불필요한 템플릿/중복 문구가 없는가?
- [ ] 오탈자 및 Markdown 렌더링 문제가 없는가?

## 7. 파일 관리 규칙

- 이 가이드 파일명은 `README_write_guide.md`로 고정한다.
- README 대규모 개편 시, 이 가이드도 함께 업데이트한다.
- README 작성 기준이 바뀌면 변경 이유를 커밋 메시지 또는 PR 본문에 남긴다.
